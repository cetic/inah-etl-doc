    Introduction

    Cette documentation est destinée aux utilisateurs de l'API du service ETL et vise à fournir plus de détails pratiques sur son utilisation, les appels, les schémas et les formats de données à utiliser.

    La spécification de l'interface proprement dite est définie dans le fichier openapi.yaml et est décrite dans ce document.

    Ce document est cours de rédaction. Son contenu peut encore changer et doit être validé avant publication.

    Le service d'ETL

    Le service d'ETL (Extraction, Transformation, Loading) permet d'uploader des données dans la base de données. Actuellement, les types données supportés sont : - Act : Les actes médicaux - Disease : Les maladies - Measure : Les mesures (type mesure de laboratoire) - Prescription : Les prescriptions de médicaments et autres spécialités pharmaceutiques.

    Chaque donnée d'un de ces types est à mettre en lien avec le patient auquel elle se rapporte, et lorsque c'est possible, avec son auteur. Elles contiennent toutes un "timestamp" permettant d'horodater l'évènement de la création de la donnée.

    L'appel à ce service se fait via une interface type "REST" en appelant la méthode "POST" de la route correspondant au type de données.

    Les sections suivantes décrivent les différentes routes et la manière de les appeler.

    Les différentes routes partagent une base de schéma commune reprenant les champs suivants :

    hniss

    le champ hniss représente l'identifiant anonymisé du patient.

    birthYear

    le champ birthYear représente l'année de naissance du patient dans un calendrier grégorien. Elle est représentée sous la forme d'un nombre entier à 4 chiffres.

    gender

    Le champ gender représente le genre de naissance du patient. Il est spécifié sous la forme d'un caractère à sélectionner parmi:

    • M : Masculin
    • F : Féminin
    • X : Autre

    ts

    Le champ ts représente le timestamp qui horodate l'acte. Il s'agit idéalement du moment ou l'acte a été réalisé. Il est représenté par un epoch au format unix à envoyer sous la forme d'une string ne contenant que des chiffres.

    author

    Schéma :

        Author:
      type: object
      required:
        - code
        - codeType
      properties:
        code:
          type: string
        codeType:
          type: string


    {"type": "object", 
 "required": ["code", "codeType"], 
 "properties": {
    "code": {"type": "string"}, 
    "codeType": {"type": "string"}
 }
}

    Le champ author représente la personne qui a réalisé l'acte qui a produit la donnée (acte médical, diagnostic, test, prescription, etc...). Dans le cadre du projet inah, il s'agira le plus souvent d'un médecin.

    Il est représenté par un couple de champs - code : un code identifiant de la personne - codeType : le type de code utilisé (par ex: "inami" pour le numéro inami d'un médecin).

    items

    Le champ items contient la liste d'élements de données à stocker. Pour chaque route, ces éléments pourront prendre une forme différente qui sera décrite dans la section spécifique au type de donnée correspondant.

    Cette route permet d'uploader un batch d'actes médicaux

    HTTP Request

    POST /act

    Description des données

    Chaque acte médical est représenté par un couple de codes : - code : le code de l'acte médical - codeType : le référentiel dont est issu le code. (par ex: 'inami' pour un code inami)

    HTTP Request

    POST /disease

    Description des données

    Chaque maladie est représentée par un couple de codes : - code : le code de la maladie - codeType : le référentiel dont est issu le code. (par ex: 'icd10' pour un code ICD 10)

    Mesures

    Schéma :

        MeasureItem:
      type: object
      required:
        - code

        - value
      properties:
        code:
          type: string
        codeType:
          type: string
        value:
          type: string
        unit:
          type: string

    "MeasureItem": {"type": "object", 
    "required": ["code", "codeType", "value"],
    "properties": {
        "code": {"type": "string"}, 
        "codeType": {"type": "string"}, 
        "value": {"type": "string"}, 
        "unit": {"type": "string"}
    }
}

    Cette route permet d'uploader un batch de mesures relatives à un patient.

    HTTP Request

    POST /measure

    Description des données

    Chaque mesure est représentée par une série de champs - code : le code de la mesure - codeType : le référentiel dont est issu le code. (par ex: 'loinc' pour un code LOINC, 'albert' pour un code ALBERT, 'snomed' pour un code SNOMED) - value : la valeur de la mesure (par ex: 92) - unit: l'unité relative à la valeur (par ex: mmHg, %, mg/dl, mmol/l, ...)

    Le champ unité est optionnel mais fortement recommandé, particulièrement dans le cas ou différentes unités (ou échelles d'une même unité) sont habituellement utilisées.

    Cette route permet d'uploader un batch de prescriptions de médicaments ou d'autres spécialités pharmaceutiques.

    HTTP Request

    POST /prescription

    Description des données

    Chaque entrée d'une prescription est représentée par l'un des deux schémas:

    Soit une entrée codée sous la forme d'un couple : - code : le code de l'élément prescrit - codeType : le référentiel dont est issu le code. (par ex: 'cnk' pour un code CNK, 'atc' pour un code ATC)

    Soit une entrée en texte libre représentée par un champ nommé text qui contient le texte rédigé sur la prescription (utilisé par exemple pour la description d'une préparation magistrale).
